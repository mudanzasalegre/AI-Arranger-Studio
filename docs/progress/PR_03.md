# PR-03 - Epic 16.6 Artifact Quarantine + Takes

## Fecha

2026-06-25

## Resumen

Se implemento la cuarentena de artifacts de IA y el sistema de takes pendiente /
aceptado / rechazado. Una salida de modelo ya no modifica el proyecto activo de
forma directa: se guarda como artifact raw, se importa, se fusiona contra una
copia temporal, se valida, y solo puede llegar al proyecto activo si el usuario
acepta la take.

La API ya expone:

```text
GET  /v1/projects/{id}/takes
POST /v1/projects/{id}/takes/{take_id}/accept
POST /v1/projects/{id}/takes/{take_id}/reject
```

## Funcionalidad anadida

- `ArtifactStore`:
  - crea `raw`, `imported`, `validated`, `rejected`;
  - guarda artifacts generados por backends en cuarentena `raw`;
  - mantiene `artifact_manifest.json`;
  - mueve/actualiza estado a `imported`, `validated` o `rejected`.
- `ArtifactImporter`:
  - importa artifacts MIDI con `mido`;
  - convierte el contenido en un `Track` parcial limitado a pista/compases;
  - rellena silencios para preservar duraciones de compas;
  - rechaza artifacts MIDI/JSON invalidos con error controlado.
- `ProjectMerger`:
  - fusiona material importado sobre una copia del proyecto;
  - sustituye solo la pista y los compases solicitados;
  - preserva el `song_plan` y el resto del arreglo.
- `ValidationGate`:
  - ejecuta `validate_project`;
  - verifica que no cambien pistas fuera del target;
  - verifica que no cambien compases fuera del target;
  - soporta `locked_tracks`;
  - exige material musical en el rango objetivo.
- `TakeManager`:
  - crea `take_base` como snapshot aceptado inicial;
  - crea takes pendientes con snapshot, artifacts y validation report;
  - acepta takes escribiendo `arrangement_project.json`;
  - rechaza takes sin tocar el proyecto activo;
  - permite restaurar la base aceptando `take_base`.
- API:
  - listado de takes por proyecto;
  - accept/reject por take;
  - error controlado si falta el proyecto o la take.

## Flujo validado

```text
modelo genera artifact
-> ArtifactStore guarda raw
-> ArtifactImporter importa a representacion interna
-> ProjectMerger fusiona en copia temporal
-> ValidationGate valida la copia
-> TakeManager crea take pending
-> accept escribe el proyecto activo
-> reject no modifica el proyecto activo
```

## Smoke de PR-03

- Artifact mock valido crea una take pendiente.
- Artifact mock invalido queda en `rejected`.
- El proyecto activo no cambia antes de `accept`.
- `reject` no toca `arrangement_project.json`.
- `accept` escribe el snapshot candidato como proyecto activo.
- `accept take_base` restaura el snapshot base.

## Notas de implementacion

- `ArtifactStore` usa `ModelGenerationResult` solo como tipo bajo `TYPE_CHECKING`;
  asi los scripts existentes pueden importar `arranger_core` aunque
  `model_backends` no este instalado en editable.
- `ArtifactImporter` escribe en `artifact_store.root/imported` cuando recibe un
  store temporal, evitando contaminar `outputs/model_artifacts` durante tests.
- La cuarentena global queda preparada en `outputs/model_artifacts`, pero los
  tests usan roots temporales.

## Riesgos pendientes

- Todavia no hay endpoints de generacion IA que creen takes automaticamente;
  eso empieza en PR-05/PR-06.
- La validacion musical de material importado es estructural y de rango; las
  restricciones armonicas fuertes llegaran con los PRs de motores y backends.
- No se ha implementado UI de takes; queda para PR-14.
